How It Started

The Pirates of the Coral-bean hackathon, organized by WeMakeDevs and Kunal Kushwaha, kicked off May 25th with a simple premise: build agents using Coral — an open-source tool that lets you query any API, database, or file as if it were a SQL table.

I was exploring Track 2 (Personal Agent), but when I discovered the hackathon's Chart New Waters challenge — build new source specs, get them accepted by the Coral team, and earn bounties for each one — I knew exactly where to put my energy. Every new source grows the reef. So I started building.

One contribution turned into two. Two turned into sixteen.

What Is Coral?

Coral gives you a single SQL interface over any API, database, or file. Instead of writing custom fetch logic, pagination handlers, and auth wrappers for every API you need, you define a manifest.yaml — and Coral handles everything from auth to rate limiting to cross-source joins.

The magic is that sources join like database tables:

SELECT w.title, w.extract, m.artist_name
FROM wikipedia.pages w
JOIN musicbrainz.artists m ON m.name = w.title
WHERE w.title = 'David Bowie';

No ETL. No glue code. Just SQL across two live APIs. That's what I spent this week building out.

what the AI did — and where it stopped

What the AI Did — and Where It Stopped

A Coral source spec is a manifest.yaml file. You define the endpoint, auth method, tables, columns, and pagination strategy — and Coral maps everything to SQL.

Here's a simplified example from the n8n source:

dsl_version: 3
name: n8n
backend: http
base_url: https://your-n8n-instance.com/api/v1
tables:
  - name: workflows
    path: /workflows
    method: GET
    columns:
      - name: id
        type: Utf8
        from: id
      - name: name
        type: Utf8
        from: name
      - name: active
        type: Boolean
        from: active

Here's exactly what Antigravity did well: given an API documentation URL, the existing source specs as reference, and strict rules about the Coral manifest schema, it drafted the initial YAML quickly. Boilerplate column definitions that would take me an hour took minutes.

Here's where it stopped cold:

  • It didn't decide which APIs to build or why they'd be valuable to the ecosystem
  • It couldn't tell when a type choice would silently break downstream SQL queries
  • It didn't understand the difference between an API that paginates via query string vs. one that embeds the page number in the URL path
  • When maintainers left review comments, the AI couldn't interpret why something was wrong — it could only act on instructions I gave it after I understood the feedback myself

Every review cycle required me to read the feedback, sit with it, understand the architecture, and then decide how to fix it. The AI was a fast drafting tool. The judgment was mine.

the real education

What the Code Reviews Actually Taught Me

James Summerfield, Coral's CTO, reviewed my PRs during the middle of a hackathon sprint. His feedback was specific, direct, and always came with a reason. That's rarer than it sounds.

Lesson 01

Types aren't labels — they define how SQL behaves

Early on I was choosing column types based on what the API returned. If the API sent back a string, I'd store it as Utf8. Logical, right? James flagged this immediately on a source where I stored a numeric field as Utf8. With that type, ORDER BY score ASC doesn't sort numerically — it sorts lexicographically. So 10, 100, 2, 20, 9 — no error, no warning, just silently wrong results. The fix is Float64. The lesson is to think from "what does SQL behavior look like for this type", not just "what format does the API use".

Lesson 02

The gap between what a framework expects and what an API does

Some APIs paginate via a page number in the query string (?page=2). Others embed it in the URL path (/results/2). Coral's default pagination assumes query string — so path-based APIs break with a 400 Bad Request because they reject unrecognized query parameters. The solution required reading Coral's pagination implementation directly and understanding what it appends and when. That kind of problem doesn't exist in tutorials — it only appears when you're integrating a real API against a real framework.

Lesson 03

Schema hygiene at scale — YAML anchors

For sources with multiple tables sharing the same column structure, I was copy-pasting the schema. James pointed out that this creates drift — if one table gets updated and the other doesn't, the source becomes inconsistent. The fix:

tables:
  - name: table_a
    columns: &shared_columns
      - name: id
        type: Int64
      - name: name
        type: Utf8

  - name: table_b
    columns: *shared_columns

One definition. One update point. I knew DRY as a code principle. I'd never applied it to configuration until someone maintaining real infrastructure showed me why it matters.

the honest version

On Using AI — The Honest Version

Early in the sprint, I was moving fast but understanding less. I was prompting the agent, running coral source lint, and submitting. Then the review comments started.

Each one forced me to slow down and understand why the code was wrong — not just apply the fix and move on. That's where the real learning happened. Not in the AI output, but in the gap between what the AI produced and what an experienced maintainer said it should be.

Using AI without understanding the output isn't contributing — it's copy-pasting with extra steps.

If you're using AI for open-source contribution: use it to implement faster. But own every decision it makes. If a maintainer asks why you chose Utf8 over Int64 for a numeric field, the AI can't answer for you. That answer has to come from your own understanding.

A Note on the Sources Themselves

Looking back at the list, the range makes me proud. Auth0, n8n, and Trello are real enterprise tools that teams use every day — and they're now SQL tables. OpenAlex gives researchers a way to query 250 million academic papers without touching an API client. MusicBrainz makes music metadata searchable in a way it wasn't before in Coral.

And then there's Open Notify. The fact that you can now write:

SELECT latitude, longitude, timestamp
FROM open_notify.iss_position;

...and get the live location of the International Space Station in your terminal — that's what an open data ecosystem can actually be.
